# 知识库维护说明

本目录是技术标写作时检索引用的通用工程知识库（跨项目共享），和"项目内知识条目"
（每个项目单独上传的参考标书/资料，走 `KnowledgeItem` 表，在项目详情页里管理）是
两回事，不要混淆：

| | 本目录 `knowledge/` | 项目内知识条目 |
|---|---|---|
| 范围 | 跨项目共享的通用工艺知识 | 单个项目专属的参考资料 |
| 存放形式 | txt 文件 + git 版本管理 | 数据库 `KnowledgeItem` 表 |
| 管理方式 | 直接编辑文件 / `import_knowledge.py` | 项目页面「知识库」抽屉里传文件 |
| 谁能看到 | 所有匹配该分类的项目 | 只有对应的那个项目 |

---

## 1. 目录结构

```
knowledge/
  _registry.yaml          # 唯一需要维护的"注册表"，决定每类项目能看到哪些文件夹
  主变安装/
    xxx.txt
  GIS安装/
    xxx.txt
  ...
```

一个文件夹 = 一个知识分类，通常对应大纲里的一个主要施工方案章节（如"GIS组合电器
安装方案""质量保证措施"）。文件夹名就是代码里的 `bound_folder`，会直接展示在检索
日志和降级提示里，起名要和大纲章节标题风格一致。

## 2. 文件内容格式

每个 `.txt` 文件里，**空行分隔的每一段**就是检索的最小单元（chunk）。建议每段
300-800 字，太短检索不到细节，太长会稀释匹配度、还容易被截断进 prompt 时挤掉别的
内容。

每段可以在正文前加两行元数据（可选，没有也兼容）：

```
## 关键词: 本体吊装, 气室对接, SF6气体
## 适用章节: GIS组合电器安装方案, 质量保证措施
本体吊装前应完成基础验收，采用专用吊具进行整体就位……（正文）
```

- `## 关键词:` 和 `## 适用章节:` 里的词会参与 BM25 检索匹配（不参与语义向量），
  写清楚能明显提高该段被命中的概率，尤其是正文里没直接出现、但用户查询时可能用到
  的同义说法。
- 正文本身不要出现具体项目名称、中标单位、金额等信息——这是通用知识库，会被所有
  匹配分类的项目共享检索到。
- 不写元数据也没问题，就是普通一段文本，照样能被检索到，只是关键词匹配会弱一些。

## 3. 增加一个新的知识分类

1. `knowledge/` 下新建文件夹，比如 `knowledge/质量保证措施/`
2. 放一个或多个 `.txt` 文件进去，按第 2 节的格式写内容
3. 打开 `knowledge/_registry.yaml`，把文件夹名加进对应列表：
   - `default`：仅电力工程在类型未命中时使用
   - `线路工程` / `电缆工程` 等：电力子类型
   - 或直接用工程领域 key（如 `市政工程:`）登记该领域可用文件夹
   - 非电力领域若未登记，系统返回空列表（不会误绑电力知识）
4. 完事——不需要改任何 `.py` 文件，不需要重启服务（`_registry.yaml` 是按修改时间
   自动重新加载的）

## 4. 修改 / 删除一个知识分类

- **改文件夹名**：直接改文件夹名，同步把 `_registry.yaml` 里对应的名字也改掉。
  两处不一致的话，运行 `pytest tests/test_knowledge_registry.py` 会直接报错提醒你。
- **删除**：删掉文件夹，同时把 `_registry.yaml` 里的引用也删掉。如果只删文件夹忘
  了删 registry 引用，检索时会查到空目录（走"知识库为空"的降级提示，不会报错，
  但相当于白白浪费了一次绑定机会，最好还是及时清理）。
- 改动会在下次该分类被检索时自动同步进数据库（新增的重新分片入库、消失的段落
  自动清理），不需要手动清库。

## 5. 批量导入历史素材

手头有历史中标标书片段、工艺手册这类原始长文本时，用：

```bash
python scripts/import_knowledge.py knowledge_source/某文件.txt --folder 质量保证措施 --dry-run
```

会调用 LLM 把长文切成符合第 2 节格式的多个片段，`--dry-run` 先预览前两条，确认没
问题再去掉这个参数正式写入。脚本会自动跳过和已有内容重复的段落。

> 注意：目前单次处理的原文长度上限是 6000 字，超过会被截断，暂时不支持自动分批。
> 素材比较长的话，先手动切成几段分别跑。

## 6. 内容来源要求

这里存的必须是**真实的工艺知识**（来源可以是历史中标标书、企业内部工艺手册、
国家/行业标准里的通用做法），不要让 AI 凭空编。检索基础设施再完善，喂进去的是
编造的内容，最后写出来的标书就是编造的内容包了一层"看起来很像检索出来的"外壳，
反而更难发现问题。

## 7. 自检

改完知识库结构后，跑一下：

```bash
pytest tests/test_knowledge_registry.py tests/test_retrieval_service.py
```

`test_knowledge_registry.py` 会检查 `_registry.yaml` 里登记的文件夹是否都真实存在、
磁盘上是否有文件夹没登记进任何 registry 分类（这类文件夹永远不会被绑定到章节，等
于白建）。
